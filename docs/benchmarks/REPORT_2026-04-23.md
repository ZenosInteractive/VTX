## 2. How to read the output

- **Time**: wall-clock per iteration.
- **CPU**: on-core time per iteration (lower than wall when the iter waits on I/O or another thread — typical for any benchmark that opens files, schedules prefetches, or serialises to disk).
- **Iterations**: google/benchmark runs the body until ≥0.5 s is accumulated, then reports the per-iteration time. Single-iter scenarios are heavy open-and-scan workloads where a half-second budget only fits one pass.
- **items_per_second**: normalised throughput. Denominator is CPU time, not wall time — important to keep in mind when wall ≫ CPU (async prefetch paths).
- **ns_per_frame**: convenience counter for per-frame cost on sequential scans.

### 4.3 Accessor layer — three distinct API surfaces

The SDK exposes property access in three layers. Each has its own cost profile; the benchmark suite measures them separately:

- **`FrameAccessor`** (`reader->CreateAccessor()`) — top-level object that knows the replay's schema and can turn `(struct_name, prop_name)` pairs into typed handles. **Created once per integration.**
- **`PropertyKey<T>`** (`accessor.Get<T>("Struct","prop")`) — pre-resolved typed handle. **Resolved once per property an integration cares about.**
- **`EntityView`** (`EntityView view(entity); view.Get(key)`) — ad-hoc wrapper over a raw entity, constructed inside the hot loop to read a typed value.

#### 3.3.1 Isolated per-layer cost (arena fixture, 3 600 frames, ~1.8 MB)

| Layer | Benchmark | Cost | Per-op derivation | Interpretation |
|---|---|---:|---|---|
| FrameAccessor creation | `BM_FrameAccessor_Creation` | **6.77 µs** | one call per iter | One-time cost after opening a replay. Schema-driven handshake. |
| PropertyKey resolution | `BM_AccessorKeyResolution` | **0.66 µs / 9 keys** | ~**74 ns per key** | Setup cost paid once per property an integration needs. |
| `EntityView` + `Get` (isolated) | `BM_EntityView_SingleGet` | **1.70 ns** (585 M/s) | one `EntityView` + one `Get` per iter | Steady-state per-property-read cost. Theoretical floor. |

These three benchmarks answer "what does a consumer actually pay for the accessor API?" in layer-separated form, which is the framing sales / DevRel need. They're new in this run; the earlier run only had the bundled `BM_AccessorSequentialScan`.

#### 3.3.2 Hot-loop costs (multi-entity, all-RAM)

| Benchmark | Time/iter | CPU/iter | items/s | Notes |
|---|---:|---:|---:|---|
| `BM_AccessorHotLoopPreloaded` | 2.81 ms | 2.78 ms | 12.9 M | All frames pre-loaded; sequential iteration; `EntityView::Get` per Player. Mixes entity iteration overhead into the per-read cost. |
| `BM_AccessorRandomWithinBucket` | 1.16 ms | 1.19 ms | 60.7 M ⚠️ | Shuffled entity indices within each bucket. **Counter inflated ~2× by a duplicate-push in the fixture setup — see footnote.** Signal is "shuffled access still fits in cache" rather than a clean number. |
| `BM_AccessorSequentialScan` | 83.7 ms | 23.1 ms | 156 k | End-to-end: open file + CreateAccessor + resolve keys + enumerate frames + EntityView + read. The catch-all "integration cost for one full replay walk" number. |

> ⚠️ **Known issue in `BM_AccessorRandomWithinBucket`**: the fixture-prep loop pushes each Player entity twice into the shuffle vector, so `ops_per_sweep` (and therefore `items_per_second`) reads 2× the real unique-entity count. Flagged as a separate task; will be fixed in a follow-up. Until then, this row's throughput number is not directly comparable with `BM_AccessorHotLoopPreloaded`.

#### 3.3.3 Reading the isolated numbers in context

For a consumer integration:

- **Startup cost**: ~7 µs (FrameAccessor) + ~74 ns × N (PropertyKey per property) = **~8 µs for a typical 10-property integration**, once. Invisible.
- **Steady-state read cost**: **1.70 ns per `EntityView::Get`** when every cache is warm. That's the ceiling for what any hot loop can achieve.
- **Real-world read cost** in a loop that also iterates entities: 52–220 ns per read depending on access pattern and cache behaviour. The gap between 1.70 ns and ~52 ns is the overhead of the surrounding entity-iteration loop, not of the accessor layer itself.

### 4.5 CS cache-window sweep — `BM_CS_AccessorRandomAccess_CacheSweep_FBS`

50 random accesses per iter, symmetric cache window `SetCacheWindow(N, N)`:

| Cache window | Wall time | items/s | Observation |
|---:|---:|---:|---|
| 0 | 14.56 s | 25.2 /s | Cold baseline. Every access pays a chunk load. |
| 2 | 23.20 s | 8.2 /s | **1.6× SLOWER than cold.** Thrash zone. |
| 5 | 16.87 s | 6.7 /s | Still thrashing. |
| 10 | 3.27 s | 17.7 /s | All cache hits (fixture fits fully). |
| 20 | 3.09 s | 18.9 /s | Plateau — overshoot doesn't help. |

**This is the headline finding for consumers.** "Some cache is better than no cache" is false here: a too-small symmetric window evicts recently-loaded chunks before they amortise, so every random jump pays the eviction cost *plus* the load cost. The CS fixture has ~10 chunks, so a window ≥10 flips into all-cache-hits territory and throughput jumps 5×. Guidance for SDK users: **size the window to the expected scrubbing span, not arbitrarily small**.

## 5. Format comparison: FlatBuffers vs Protobuf

One fixture at a time, same access pattern. Using **median** (runs with outliers noted in §4.4):

| Fixture | FBS median seq-scan | Proto median seq-scan | Winner | Margin |
|---|---:|---:|---:|---:|
| CS (92/83 MB, 10 656 frames) | 5560 ms | 2934 ms | **Proto** | FBS 90 % slower |
| RL (4.8/5.1 MB, ~21 k frames) | **723 ms** | 2556 ms | **FBS** | Proto 3.5× slower |

**The format winner is fixture-shape dependent.** This is worth flagging to consumers who ask "which backend is faster?" — the honest answer is "measure on *your* replay shape". CS has large per-frame payloads that hit FBS's random-access overhead harder than Proto's streaming decode; RL has a different schema density that favours FBS's zero-copy.

## 6. Key findings for the team

1. **Cache-window sweep confirms the "middle-window trap".** Random-access workloads on CS-sized replays are **faster with no cache** than with a small-but-not-covering window. Recommendation: document in user-facing guidance that `SetCacheWindow` should size to the expected scrub span, not a small default.
2. **Accessor layers now measured separately.** `FrameAccessor` creation (6.77 µs), `PropertyKey` resolution (74 ns/key), and `EntityView::Get` (1.70 ns) are each benchmarked in isolation. The steady-state read ceiling is **585 M reads/s**. Useful for talking to consumers about their expected hot-loop cost.
3. **CS sequential-scan variance is structural** — the outlier moves between FBS and Proto across runs, so it's not a format-specific issue. Likely OS page-cache warm-up on the first iteration of a 92 MB scan. Needs wider `repeats` or longer `benchmark_min_time` to stabilise.
4. **FBS vs Proto is fixture-dependent.** Proto is ~90 % faster than FBS on CS median; FBS is 3.5× faster than Proto on RL. Any README claim of a single "faster backend" needs qualification.
5. **Differ short-circuit is healthy** — 10× on small frames, ~2× on CS. The xxHash short-circuit earns its keep.
6. **Known fixture bug**: `BM_AccessorRandomWithinBucket` duplicates Player indices in its shuffle setup, inflating `items_per_second` by 2×. Spawned as a follow-up task.

## 7. Caveats

- Single-host, single-run. Wall-time variance in the 5 %–10 % range is expected on Windows; the CS outliers are a bigger concern (see §4.4).
- Windows background activity (antivirus, Defender real-time) was not controlled. Wall times for open-heavy benchmarks (`WarmFileCache` notwithstanding) are soft upper bounds.
- No Linux Release comparison in this run. Worth rerunning on CI's Linux runner to separate OS-page-cache behaviour from Windows.
- google/benchmark's `items_per_second` uses **CPU time** as denominator. Where wall ≫ CPU (CS random access, cache-sweep cold), the items/s figure overstates user-observable throughput. Use the wall-time column for product-facing claims.

## 8. Reproduce

```bash
# Configure (Release, benchmarks on)
cmake -S . -B build-bench -DCMAKE_BUILD_TYPE=Release -DVTX_BUILD_BENCHMARKS=ON

# Build
cmake --build build-bench --target vtx_benchmarks --config Release --parallel

# Run full suite, emit JSON + console
build-bench/bin/Release/vtx_benchmarks.exe \
    --benchmark_out=docs/benchmarks/bench_$(date +%Y%m%d_%H%M%S).json \
    --benchmark_out_format=json \
    --benchmark_counters_tabular=true | tee docs/benchmarks/bench_$(date +%Y%m%d_%H%M%S).txt

# Only the isolated accessor layers
build-bench/bin/Release/vtx_benchmarks.exe \
    --benchmark_filter='BM_FrameAccessor_Creation|BM_EntityView_SingleGet|BM_AccessorKeyResolution'

# Only the cache-window study
build-bench/bin/Release/vtx_benchmarks.exe \
    --benchmark_filter='BM_CS_AccessorRandomAccess_CacheSweep_FBS'

# Requires CS/RL/arena fixtures under samples/content/reader/{cs,rl,arena}/
# and the synth_10k.vtx fixture under benchmarks/fixtures/ (generated at
# build time by vtx_sample_write when VTX_BUILD_BENCHMARKS=ON).
```
